Overview

The ESTLM Application uses a versatile set of table components to display and manage student absence information and attendance records. These data tables are designed to be responsive, interactive, and capable of displaying complex student and absence data in a clear, structured format.

Key Table Component Features

  • Modular Structure - Composed of individual components that can be combined as needed
  • Responsive Design - Tables adapt to different screen sizes
  • Semantic Markup - Uses proper HTML table elements for accessibility
  • Visual Feedback - Includes hover states and status indicators
  • Filtering - Built-in search and filtering capabilities
  • Status Visualization - Color-coded badges for status indication

Implementation

The table implementation in the ESTLM application is based on a set of modular React components. The base table components are defined in client/src/components/ui/table.tsx and follow a compositional pattern where each part of the table (header, body, row, cell) is defined as a separate component.

Usage Examples

The table components are used throughout the application to display different types of data. Below are examples of how these components are used in specific contexts.

Absence Reports Table

The Absence Reports Table displays a list of students with their absence information, including total absence hours and status indicators. It also includes filtering capabilities and action buttons. 

// From AbsenceReports.jsx
import React, { useState } from 'react';
import { Table, TableBody, TableCell, TableHead, TableRow } from './TableComponents';
import { Badge } from './Badge';
import { Button } from './Button';

const AbsenceReportsTable = ({ reports }) => {
  const [searchQuery, setSearchQuery] = useState('');

  const filteredReports = reports.filter((report) =>
    report.studentName.toLowerCase().includes(searchQuery.toLowerCase())
  );

  return (
    <div>
      <input
        type="text"
        placeholder="Search by student name"
        value={searchQuery}
        onChange={(e) => setSearchQuery(e.target.value)}
        className="mb-4 p-2 border rounded"
      />
      <Table>
        <TableHead>
          <TableRow>
            <TableCell>Student Name</TableCell>
            <TableCell>Student Identifier</TableCell>
            <TableCell>Department</TableCell>
            <TableCell>Semester</TableCell>
            <TableCell className="text-center">Total Absence Hours</TableCell>
            <TableCell>Status</TableCell>
            <TableCell className="text-right">Actions</TableCell>
          </TableRow>
        </TableHead>
        <TableBody>
          {filteredReports.length > 0 ? (
            filteredReports.map((report) => (
              <TableRow key={report.studentId}>
                <TableCell className="font-medium">{report.studentName}</TableCell>
                <TableCell>{report.studentIdentifier}</TableCell>
                <TableCell>{report.departmentName}</TableCell>
                <TableCell>{report.semesterName}</TableCell>
                <TableCell className="text-center">{report.totalAbsenceHours}</TableCell>
                <TableCell>
                  <Badge
                    variant={report.totalAbsenceHours > 9 ? 'destructive' : 'secondary'}
                  >
                    {report.totalAbsenceHours > 9 ? 'Exceeded Limit' : 'Normal'}
                  </Badge>
                </TableCell>
                <TableCell className="text-right">
                  <Button
                    variant="ghost"
                    size="sm"
                    onClick={() => handleViewDetails(report.studentId)}
                  >
                    View Details
                  </Button>
                </TableCell>
              </TableRow>
            ))
          ) : (
            <TableRow>
              <TableCell colSpan={7} className="h-24 text-center">
                {searchQuery ? 'No reports match your search.' : 'No absence reports found.'}
              </TableCell>
            </TableRow>
          )}
        </TableBody>
      </Table>
    </div>
  );
};

Best Practices

When working with the table components in the ESTLM application, follow these best practices:

Structural Guidelines

  • Always use semantic structure - Include proper TableHeader, TableBody, TableHead, etc. elements for accessibility
  • Provide captions - Use TableCaption to describe the purpose and content of tables
  • Include empty state handling - Always handle the case when no data is available
  • Consistent column widths - Set appropriate column widths to prevent layout shifts
  • Responsive considerations - For mobile, consider which columns can be hidden or collapsed

Interaction Guidelines

  • Search and filtering - For large tables, always provide search functionality
  • Status indicators - Use badges for status to provide visual cues
  • Action buttons - Place consistent action buttons in a dedicated column
  • Sorting - Support column sorting for tables with many rows
  • Use hover effects - Apply hover effects to rows to aid in tracking

Visual Design Guidelines

  • Zebra striping - Use alternating row colors to aid readability in larger tables
  • Emphasis for important data - Use font-weight to emphasize primary identifiers
  • Color-coding thresholds - Use consistent color coding for status indicators (green/yellow/red)
  • Text alignment - Left-align text, center numeric values, and right-align actions
  • Consistent padding - Apply consistent padding to cells for better readability

Accessibility Considerations

The data tables in the ESTLM application are built with accessibility in mind:

  • Semantic HTML - Uses proper table elements (table, thead, tbody, th) for screen readers
  • Keyboard navigation - Supports standard keyboard navigation through rows and cells
  • Color contrast - Maintains sufficient contrast between text and background
  • Text alternatives - Includes clear text alternatives for visual indicators
  • Descriptive captions - Uses TableCaption to describe the table contents
  • Responsive design - Tables adapt to different screen sizes and zoom levels
